## Directory Purposes

**`.github/workflows/`:**
- Purpose: Continuous integration configuration
- Contains: GitHub Actions workflow that builds firmware on push/PR
- Key files: `build.yml` - references ZMK's shared build workflow

**`config/`:**
- Purpose: User keyboard configuration (root namespace for ZMK)
- Contains: Global keymaps, settings, and board/shield definitions
- Key files: `charybdis.keymap`, `charybdis.conf`, `west.yml`

**`config/boards/shields/charybdis/`:**
- Purpose: Hardware and device tree definitions for Charybdis keyboard
- Contains: GPIO mappings, sensor configuration, keymap transforms, left/right half differences
- Key files: `charybdis.dtsi` (base), `charybdis_left.overlay` (left), `charybdis_right.overlay` (right)

**`.planning/codebase/`:**
- Purpose: Architecture and structure documentation
- Generated by: GSD mapping tools
- Used by: GSD planning/execution tools for context

## Key File Locations

**Entry Points:**

**`build.yaml`:**
- Purpose: Defines which board/shield combinations to build
- Format: YAML array of board + shield pairs
- Current targets: nice_nano with charybdis_left/charybdis_right/settings_reset
- Usage: GitHub Actions reads this to determine what firmware to compile

**`config/west.yml`:**
- Purpose: Dependency resolution and source organization
- Declares: ZMK firmware (zmkfirmware/zmk@main), PMW3610 driver (badjeff/zmk-pmw3610-driver@main)
- Usage: `west init` clones dependencies; `west update` fetches changes

**Configuration Files:**

**`config/charybdis.conf`:**
- Purpose: Global keyboard configuration flags
- Content: Bluetooth TX power, BLE experimental features, pointing module enable, behavior queue size
- Format: Kconfig format (`CONFIG_KEY=value`)

**`config/boards/shields/charybdis/charybdis.conf`:**
- Purpose: Empty placeholder
- Usage: Can be used for future shield-wide configuration

**`config/boards/shields/charybdis/charybdis_right.conf`:**
- Purpose: Right half specific configuration
- Content: SPI driver enable, INPUT driver enable, pointing module, PMW3610 driver enable, CPI, power saving parameters
- Format: Kconfig format

**`config/boards/shields/charybdis/Kconfig.defconfig`:**
- Purpose: Automatically applied defaults when charybdis shield is selected
- Content: Enables ZMK_SPLIT, sets BLE central role for right half, sets keyboard name
- Format: Kconfig conditional syntax

**Core Logic:**

**`config/charybdis.keymap`:**
- Purpose: Define all keyboard behavior (key mappings, layers, combos, custom behaviors)
- Lines 1-9: Include directives for ZMK data bindings and behaviors
- Lines 10-11: Macro definition for tapping term (200ms)
- Lines 12-27: Trackball listener (input processor definitions, scroll layer override)
- Lines 29-71: Combos (Delete, Backspace, Emoji, GUI, Enter, Mac/Win toggle, brackets)
- Lines 73-101: Custom behaviors (u_mt, u_lt, Shift_Enter)
- Lines 103-159: Keymap layers (6 layers: BASE, SwapCtrlGui, NUM, NAV, FUN, SCROLL)
- Format: Device tree source (`.dtsi`)

**`config/boards/shields/charybdis/charybdis.dtsi`:**
- Purpose: Shared hardware definitions for both left and right halves
- Lines 1-8: Battery voltage monitor declaration
- Lines 10-21: Default keymap transform (6-column × 4-row layout)
- Lines 23-34: Five-column transform variant
- Lines 36-47: Keyboard matrix scanner (kscan0) with GPIO row definitions
- Format: Device tree source (`.dtsi`)

**`config/boards/shields/charybdis/charybdis_left.overlay`:**
- Purpose: Left half-specific overrides
- Lines 6-8: Include base hardware and trackball definitions
- Lines 11-20: Override kscan0 with left-half column GPIOs
- Lines 22-28: Disable SPI interface (trackball disabled on left)
- Format: Device tree overlay (`.overlay`)

**`config/boards/shields/charybdis/charybdis_right.overlay`:**
- Purpose: Right half-specific overrides
- Lines 7-10: Include base hardware, trackball, and input event bindings
- Lines 13-19: Override default_transform column offset (6 columns for right half)
- Lines 17-19: Override five_column_transform column offset (5 columns for right half)
- Lines 21-30: Override kscan0 with right-half column GPIOs
- Format: Device tree overlay (`.overlay`)

**`config/boards/shields/charybdis/pmw3610.dtsi`:**
- Purpose: PMW3610 trackball sensor hardware configuration
- Lines 1-12: Input event type and coordinate definitions (relative motion, wheel)
- Lines 14-31: SPI pinctrl definitions (clock, MOSI, MISO pins)
- Lines 33-60: SPI0 peripheral setup and trackball device definition (CPI, orientation, IRQ pin)
- Format: Device tree source (`.dtsi`)

**Hardware Definition:**

**`config/boards/shields/charybdis/charybdis-layouts.dtsi`:**
- Purpose: Physical key layout in x-y coordinates (mm)
- Lines 1-53: Physical_layout0 defining 41 keys with positions (width, height, x, y, rotation)
- Used by: ZMK Studio for visual keymap editing
- Format: Device tree source with physical_layout compatible

**`config/info.json`:**
- Purpose: Keyboard metadata for ZMK Studio
- Content: Keyboard ID, name, layout definition (key positions in coordinates), sensor declarations
- Format: JSON

**Metadata:**

**`config/boards/shields/charybdis/charybdis.zmk.yml`:**
- Purpose: Shield metadata declaration
- Content: Shield ID, name, URL, feature list (keys, pointer), left/right sibling shields
- Usage: ZMK build system discovers shield capabilities

**`config/boards/shields/charybdis/Kconfig.shield`:**
- Purpose: Kconfig identifiers for conditional compilation
- Declares: SHIELD_CHARYBDIS_LEFT, SHIELD_CHARYBDIS_RIGHT markers
- Usage: Kconfig uses these to conditionally apply build flags

**Testing:**
- No dedicated test directory
- No test files present

## Naming Conventions

**Files:**

**Keymap files:**
- Pattern: `[keyboard-name].keymap`
- Example: `charybdis.keymap`
- Format: Device tree source with `.keymap` extension

**Configuration files:**
- Pattern: `[context].conf` (global) or `[shield-name]_[side].conf` (side-specific)
- Examples: `charybdis.conf`, `charybdis_right.conf`
- Format: Kconfig text format

**Device tree files:**
- Pattern: `[shield-name].dtsi` (base shared), `[shield-name]_[side].overlay` (side-specific)
- Examples: `charybdis.dtsi`, `charybdis_left.overlay`, `charybdis_right.overlay`, `pmw3610.dtsi`
- Format: Device tree source (`.dtsi`) or overlay (`.overlay`)

**Device tree includes:**
- Pattern: `[hardware-component].dtsi`
- Examples: `charybdis-layouts.dtsi`, `pmw3610.dtsi`
- Format: Device tree source

**Build/manifest files:**
- Pattern: standard ZMK names
- Examples: `build.yaml`, `west.yml`
- Format: YAML

**Directories:**

**Shield directories:**
- Pattern: `boards/shields/[shield-id]/`
- Example: `boards/shields/charybdis/`

**Workflow directories:**
- Pattern: `.github/workflows/`
- Standard GitHub Actions location

## Where to Add New Code

**New Key Binding / Layer:**
- Primary code: `config/charybdis.keymap` (add to `keymap` section or `behaviors` section)
- Related: May need combo definitions in `combos` section or input processor additions in `trackball_listener`
- Pattern: Follow existing layer structure (4 rows × 12 columns binding grid)

**New Hardware Feature (Sensor, LED, etc.):**
- Base definition: `config/boards/shields/charybdis/charybdis.dtsi`
- Side-specific overrides: `config/boards/shields/charybdis/charybdis_left.overlay` or `charybdis_right.overlay`
- Pin configuration: `config/boards/shields/charybdis/[component-name].dtsi`
- Example pattern: PMW3610 trackball is defined in `pmw3610.dtsi` and included in overlays

**New Configuration Parameter:**
- Global setting: `config/charybdis.conf`
- Side-specific setting: `config/boards/shields/charybdis/charybdis_right.conf` or `charybdis_left.conf`
- Conditional defaults: `config/boards/shields/charybdis/Kconfig.defconfig`
- Format: `CONFIG_FEATURE_NAME=y` for boolean, `CONFIG_PARAM=123` for numeric

**New Custom Behavior:**
- Location: `config/charybdis.keymap`, in `behaviors` section (lines 73-101)
- Pattern: Define `compatible = "zmk,behavior-[type]"`, set `bindings`, configure `tapping_term_ms` if hold-tap
- Reference: See `u_mt`, `u_lt`, `Shift_Enter` examples

**New Build Target (Additional Board/Shield):**
- Location: `build.yaml`
- Pattern: Add entry to `include` array with `board` and `shield` fields
- Example:
  ```yaml
  - board: nice_nano
    shield: charybdis_left
  ```

**Input Processor Chain Change:**
- Location: `config/charybdis.keymap`, in `trackball_listener` (lines 13-27)
- Pattern: Add or modify `input-processors` array
- Example: `<&zip_xy_scaler 1 2>` for scaling, `<&zip_scroll_transform>` for transformations

## Special Directories

**`config/boards/shields/charybdis/`:**
- Purpose: All hardware definitions for this specific keyboard shield
- Generated: No, user-maintained
- Committed: Yes, all files tracked in git

**`.planning/codebase/`:**
- Purpose: Generated architecture documentation
- Generated: Yes, by GSD mapping tools
- Committed: Yes, documentation tracked in git

**`.github/workflows/`:**
- Purpose: GitHub Actions CI/CD configuration
- Generated: No, user-maintained
- Committed: Yes, workflow files tracked in git

**`config/boards/shields/charybdis/` (overlays):**
- Purpose: Hardware-specific configuration that extends base definitions
- Split pattern: Left half disables trackball (`status = "disabled"`), right half enables it
- Usage: Build system loads base `.dtsi`, then applies `.overlay` patches for each side
